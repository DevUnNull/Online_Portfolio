I. Introduction & Importance I. Giới Thiệu & Tầm Quan Trọng

  • Prevent chaos: use design principles to avoid creating unmanageable code spaghetti. Ngăn ngừa hỗn độn: dùng nguyên tắc thiết kế để tránh trở thành mớ khó quản lý.
  • Balance: keep structures clear while maintaining flexibility for extension. Cân bằng: giữ cấu trúc rõ ràng đồng thời linh hoạt để mở rộng.
  • Steady progress: reduce redundant work, avoid rewriting from scratch. Tiến độ ổn định: giảm công việc không cần thiết, tránh đập đi làm lại.
Note:Lưu ý: understand the intention behind each rule to know when to follow it and when to break it intentionally. hiểu intention sau mỗi quy tắc để biết khi nào nên theo, khi nào cần phá vỡ có chủ đích.

1. Single Responsibility Principle (SRP) 1. Nguyên tắc Trách nhiệm đơn lẻ (SRP)

  • A class should have only one reason to change. Một class chỉ có một lý do để thay đổi.
  • In Unity, separate ground check logic from input controls. Trong Unity, tách logic kiểm tra mặt đất khỏi điều khiển đầu vào.
  • Practical Exception: if a class is designed solely to serve the Player or a specific object and doesn't need reuse, grouping related logic still conforms to SRP since the class only changes based on that single object. Ngoại lệ thực tế: nếu class được thiết kế chỉ phục vụ Player hoặc một đối tượng cụ thể và không cần tái sử dụng cho đối tượng khác, việc gộp các logic liên quan vẫn phù hợp với SRP vì class chỉ có một lý do thay đổi gắn với đối tượng đó.
ProblemVấn đề

Movement and Ground Check are combined, making reuse difficult.Movement và Ground Check gộp chung làm khó tái sử dụng.

SolutionGiải pháp

Use inheritance: base class checks ground, derived class handles inputs.Dùng kế thừa: lớp nền kiểm tra mặt đất, lớp dẫn xuất xử lý điều khiển.

Bad Example (SRP Violation)Ví dụ sai (vi phạm SRP)

using UnityEngine;

public class PlayerController : MonoBehaviour {
    bool grounded;
    int score;
    void Update() {
        grounded = Physics.Raycast(transform.position, Vector3.down, 0.1f);
        if (grounded && Input.GetButtonDown("Jump")) {}
        if (Input.GetKeyDown(KeyCode.P)) score++;
    }
}

Good Example (Separating Ground Check from Input)Ví dụ đúng (tách Ground Check khỏi Input)

using UnityEngine;

public class Groundable : MonoBehaviour {
    protected bool IsGrounded() {
        return Physics.Raycast(transform.position, Vector3.down, 0.1f);
    }
}

public class Controllable : Groundable {
    void Update() {
        if (IsGrounded() && Input.GetButtonDown("Jump")) { }
    }
}

public class Moveable : Groundable {
    void Update() { }
}

2. Open/Closed Principle (OCP) 2. Nguyên tắc Đóng/Mở (OCP)

  • Extend with new classes, don't modify existing working code. Mở rộng bằng lớp mới, không sửa lớp đang chạy ổn.
  • Solution: Abstraction via Interfaces or Inheritance. Giải pháp: Abstraction qua Interface hoặc Inheritance.

Bad Example (OCP Violation)Ví dụ sai (vi phạm OCP)

using UnityEngine;
public class Collector : MonoBehaviour {
    void OnTriggerEnter(Collider other) {
        if (other.GetComponent<PointsPickup>() != null) {}
        else if (other.GetComponent<HealthPickup>() != null) {}
    }
}
public class PointsPickup : MonoBehaviour {}
public class HealthPickup : MonoBehaviour {}

Good Example (Using Interface for Extension)Ví dụ đúng (dùng Interface để mở rộng)

public interface ICollectable {
    void Collect(GameObject collector);
}

using UnityEngine;
public class Collector : MonoBehaviour {
    void OnTriggerEnter(Collider other) {
        var c = other.GetComponent<ICollectable>();
        if (c != null) c.Collect(gameObject);
    }
}

public class PointsPickup : MonoBehaviour, ICollectable {
    public void Collect(GameObject collector) { }
}

public class HealthPickup : MonoBehaviour, ICollectable {
    public void Collect(GameObject collector) { }
}

3. Liskov Substitution Principle (LSP) 3. Nguyên tắc Thay thế Liskov (LSP)

  • Derived objects must be substitutable for base objects without breaking expected behavior. Đối tượng dẫn xuất phải thay thế được đối tượng cơ sở mà không phá vỡ hành vi mong đợi.
  • In Unity: all Collider types are subtypes of Collider, APIs are used identically. Trong Unity: các loại Collider đều là subtype của Collider, API dùng được như nhau.
  • Keep contracts consistent: do not change parameter meanings or return value semantics. Giữ hợp đồng nhất quán: không thay đổi ý nghĩa tham số/giá trị trả về.

Bad Example (LSP Violation)Ví dụ sai (vi phạm LSP)

public class Ellipse {
    public virtual float Width { get; set; }
    public virtual float Height { get; set; }
}
public class Circle : Ellipse {
    public override float Width { get { return base.Width; } set { base.Width = value; base.Height = value; } }
    public override float Height { get { return base.Height; } set { base.Width = value; base.Height = value; } }
}

Good ExampleVí dụ đúng

using UnityEngine;
public interface IShape { float Area(); }
public class Circle : IShape { public float Radius; public float Area() { return Mathf.PI * Radius * Radius; } }
public class EllipseShape : IShape { public float A; public float B; public float Area() { return Mathf.PI * A * B; } }

4. Interface Segregation Principle (ISP) 4. Nguyên tắc Phân tách Interface (ISP)

  • Do not force classes to implement things they do not use. Không buộc class phải implement những thứ không dùng.
  • Split into multiple small interfaces according to specific needs. Tách thành nhiều interface nhỏ theo nhu cầu cụ thể.

Bad Example (ISP Violation)Ví dụ sai (vi phạm ISP)

public interface IEnemy {
    void Damage(float amount);
    void Stun(float duration);
    void Move();
}
using UnityEngine;
public class WoodenCrateWrong : MonoBehaviour, IEnemy {
    public void Damage(float amount) {}
    public void Stun(float duration) {}
    public void Move() {}
}

Good Example (Splitting into smaller interfaces)Ví dụ đúng (tách thành nhiều interface nhỏ)

public interface IDamageable { void Damage(float amount); }
public interface IStunnable { void Stun(float duration); }

using UnityEngine;
public class WoodenCrate : MonoBehaviour, IDamageable {
    public void Damage(float amount) { }
}

5. Dependency Inversion Principle (DIP) 5. Nguyên tắc Đảo ngược Phụ thuộc (DIP)

  • High-level modules should depend on abstractions, not directly on concrete low-level implementations. High-level phụ thuộc vào abstraction, không phụ thuộc trực tiếp vào low-level cụ thể.
  • Use Dependency Injection to bind implementations at runtime. Dùng Dependency Injection để gắn implementation khi chạy.

Bad Example (DIP Violation)Ví dụ sai (vi phạm DIP)

using UnityEngine;
public class MusicPlayerWrong : MonoBehaviour {
    SpecificMusicSequencer _sequencer = new SpecificMusicSequencer();
    void Start() {
        if (!_sequencer.IsPlaying()) _sequencer.StartNewSong(new SongData());
    }
}
public class SpecificMusicSequencer {
    public void StartNewSong(SongData data) {}
    public bool IsPlaying() { return true; }
}
public struct SongData {}

Good Example (High-level depending on abstraction)Ví dụ đúng (High-level phụ thuộc vào abstraction)

public interface IMusicInteraction {
    void StartNewSong(SongData data);
    bool IsPlaying();
}

using UnityEngine;
public class MusicPlayer : MonoBehaviour {
    [SerializeField] MonoBehaviour sequencerComponent;
    IMusicInteraction _sequencer;
    void Awake() { _sequencer = (IMusicInteraction)sequencerComponent; }
    void Start() { if (!_sequencer.IsPlaying()) _sequencer.StartNewSong(new SongData()); }
}

public class SpecificMusicSequencer : MonoBehaviour, IMusicInteraction {
    public void StartNewSong(SongData data) { }
    public bool IsPlaying() { return true; }
}

public struct SongData {}

IV. Conclusion IV. Kết Luận

  • SRP to split responsibilities, easy to reuse. SRP để tách trách nhiệm, dễ tái sử dụng.
  • OCP to extend safely using new classes. OCP để mở rộng an toàn bằng lớp mới.
  • LSP for consistent substitutions. LSP để thay thế nhất quán.
  • ISP for clean interfaces matching specific needs. ISP để interface gọn, đúng nhu cầu.
  • DIP to reduce hard dependencies, making testing and substitution easy. DIP để giảm phụ thuộc cứng, dễ test và thay thế.

The goal is code that updates stably, is easy to maintain and expand. Mục tiêu là code cập nhật ổn định, dễ bảo trì và mở rộng.
